# show-priorities

Keep `context/priorities.md` aligned with project frontmatter by regenerating the `## Project dashboard` section (This week / This month / Parked / Not a priority sub-buckets) inside a protected auto-zone at the bottom of the file. Diff-and-approve; never write without confirmation. The top of the file is hand-written human notes and must be preserved verbatim.

## Inputs you read

1. All `projects/work/*/README.md` files.
2. All `projects/personal/*/README.md` files (when that tree exists).
3. The current `context/priorities.md` (for diffing and marker detection).

Do NOT read `projects/school/*/index.md`. School courses are reference wikis, not priorities. Specific school commitments (exams, assignments) live as hand-written entries outside the auto-zone in `context/priorities.md` and are preserved verbatim.

Skip any file whose frontmatter is missing or malformed. Print a warning line for each skipped file in the format:

```
WARNING: <path> has no frontmatter, skipped
WARNING: <path> has malformed deadline "<value>", skipped
```

## Today's date

Use the current date from the session. All "deadline within 7 days / 30 days" comparisons are relative to today.

## Bucket inference rules (apply per project, first match wins)

1. `status: parked` → **Parked**
2. `status: done` → omit entirely
3. `deadline` within 7 days (inclusive) OR `priority: high` → **This week**
4. `deadline` within 30 days (inclusive) OR `priority: medium` → **This month**
5. otherwise → **Not a priority right now**

Each bucket renders inside the `## Project dashboard` section as an `### H3` sub-heading.

## Ordering within a bucket

1. Deadline ascending (earliest first). Projects without a deadline sort after projects with one.
2. Priority descending: high > medium > low.
3. Alphabetical by project folder slug.

## Line format

- Project name as `[[slug]]` wikilink, where `slug` is the project folder name (e.g., `some-project`).
- Domain tag in parentheses only when the project is NOT in `projects/work/`. Work is the default and gets no tag. Personal gets `(personal)`.
- Metadata separator: ` · ` (middle dot with spaces). Include fields in this order when present:
  1. `<priority> priority` (e.g., `high priority`)
  2. `deadline <YYYY-MM-DD>`, append ` OVERDUE` if the date is in the past relative to today
  3. `blocked on <list>`, comma-separated blockers. If more than 3, show the first 3 followed by `… (+N more)` where N is the number hidden.

Example lines:

```
- **[[some-project]]** — high priority · deadline 2026-04-20 · blocked on dependency-a, dependency-b
- **[[some-personal-project]]** (personal) — medium priority · deadline 2026-05-01
```

Projects with no deadline, no blockers, and only a priority render as:

```
- **[[project-name]]** (domain-if-not-work) — <priority> priority
```

Empty buckets render their header followed by the literal line:

```
_No projects in this bucket._
```

## Auto-zone markers and shape

The zone you regenerate is delimited by these exact comment lines:

```
<!-- show-priorities:start -->
<!-- show-priorities:end -->
```

Inside the markers, the content always has this shape:

```markdown
<!-- show-priorities:start -->

## Project dashboard

_Regenerated by `/show-priorities` from project frontmatter. Do not hand-edit inside the markers._

### This week
<bullets or placeholder>

### This month
<bullets or placeholder>

### Parked
<bullets or placeholder>

### Not a priority right now
<bullets or placeholder>

<!-- show-priorities:end -->
```

You MUST NOT write anything outside these markers. Everything outside is frozen and preserved verbatim, the intro, any hand-written `## This week` / `## This month` / `## Parked` / `## Not a priority right now` sections at the top of the file, every human note.

### If the markers are missing

Print the proposed auto-zone content, then ask the user:

> "`context/priorities.md` does not contain the show-priorities markers. I'll append them to the bottom of the file as a new `## Project dashboard` section. Confirm? (y/n)"

On `y`, append the full auto-zone block (including `<!-- show-priorities:start -->`, the `## Project dashboard` section with its four `###` sub-buckets, and `<!-- show-priorities:end -->`) to the end of the file, preceded by a blank line. On `n`, exit without touching anything.

## Full flow when `/show-priorities` is invoked

1. Read all `projects/work/*/README.md` and `projects/personal/*/README.md` files. Do not read `projects/school/*/`. Collect frontmatter from what you read.
2. Skip and warn on any file missing or malformed frontmatter.
3. For each valid project, apply the bucket inference rules to assign a bucket.
4. Sort each bucket using the ordering rules above.
5. Render the auto-zone block: a `## Project dashboard` H2 heading, the explanatory blurb, then one `### <bucket name>` H3 per bucket with its bullets. Empty buckets get the `_No projects in this bucket._` placeholder line.
6. Read the current `context/priorities.md`.
7. If the auto-zone markers are missing, take the missing-marker fallback path (confirm with the user, then append to the bottom of the file).
8. Otherwise, construct the proposed new file: everything before `<!-- show-priorities:start -->`, then the start marker, then a blank line, then the rendered dashboard block, then a blank line, then the end marker, then everything after `<!-- show-priorities:end -->`.
9. Compute a unified diff between the current file and the proposed file (context of 3 lines).
10. Count the changed lines. If changed lines > 50% of the auto-zone line count, print the full proposed auto-zone instead of a diff and say "diff too large, showing full auto-zone".
11. Print any skipped-file warnings collected in step 2.
12. Print the diff (or full-zone fallback).
13. Prompt: `Apply this update? (y/n)`
14. On `y`, write the proposed content to `context/priorities.md` and confirm with `Wrote context/priorities.md`.
15. On `n`, exit without touching anything and say `No changes written`.

## Edge cases

- **No frontmatter:** skip with warning.
- **`status: done`:** omit from the auto-zone.
- **Deadline in the past:** still render in the bucket determined by the rules; append ` OVERDUE` to the deadline field.
- **Tied projects:** alphabetical by folder slug breaks the tie.
- **Empty project folder (no README):** silently skip.
- **More than 3 blockers:** truncate display, indicate hidden count.
- **Malformed `deadline` value (not YYYY-MM-DD):** skip with warning.

## What this skill does NOT do

- Does not read or write school `index.md` files.
- Does not write to project READMEs.
- Does not accept any flags or arguments.
- Does not auto-invoke on hooks or schedules.
- Does not touch content outside the auto-zone.
- Does not bucket projects with `status: done`.
